Это скорее "архитектурный набор правил", чтобы классы и объекты не превращались в кашу.

1. **S — Single Responsibility Principle (Принцип единственной ответственности)**
    
    - Каждый класс должен иметь **одну причину для изменения**.
        
    - Пример: класс «Заказ» не должен и печатать чеки, и считать скидки, и общаться с базой.
        
    - Нарушение: «God Object», который делает всё подряд.
        
2. **O — Open/Closed Principle (Открытость/закрытость)**
    
    - Класс открыт для **расширения**, но закрыт для **изменений**.
        
    - Идея: добавляем новый функционал через наследование, интерфейсы или композицию, а не переписывая старый код.
        
    - Пример: вместо «if (тип документа == PDF) else if (Word)» — используем абстрактный класс `DocumentExporter`, и добавляем новые подклассы.
        
3. **L — Liskov Substitution Principle (Принцип подстановки Лисков)**
    
    - Объект подкласса должен вести себя так же, как базовый класс, чтобы его можно было заменить.
        
    - Пример: «Круг» не должен наследоваться от «Эллипс», если это ломает математику площади.
        
4. **I — Interface Segregation Principle (Принцип разделения интерфейсов)**
    
    - Лучше несколько маленьких интерфейсов, чем один огромный.
        
    - Пример: вместо `IMachine` с методами `Print`, `Scan`, `Fax`, `Staple` — лучше `IPrinter`, `IScanner` и т.д.
        
5. **D — Dependency Inversion Principle (Принцип инверсии зависимостей)**
    
    - Зависеть нужно от **абстракций**, а не от конкретных классов.
        
    - Пример: `ReportService` должен работать с интерфейсом `ILogger`, а не с `ConsoleLogger`.
