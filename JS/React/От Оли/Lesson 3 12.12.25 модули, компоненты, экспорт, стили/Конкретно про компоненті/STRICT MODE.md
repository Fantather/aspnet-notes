Обычно `StrictMode` оборачивает всё приложение в корневом файле (`index.js` или `main.jsx`).


Это строгий режим кода. 
он помогает разрабам находить потенциальные ошибки и плохие практики еще на этапе разработки

реакт следит что бы вы использовали только современные методы разработки

### 1. Двойной вызов рендеринга (Double Invocation)

Это самое заметное поведение, которое часто пугает новичков (почему `console.log` срабатывает дважды?).

- **Что происходит:** React намеренно вызывает функции компонентов, хуки (useState, useMemo, useReducer) и методы жизненного цикла **два раза подряд**.
    
- **Зачем:** Чтобы убедиться, что ваши функции являются **чистыми** (pure).
    
    - _Чистая функция_ при одинаковых входных данных всегда возвращает одинаковый результат и не меняет ничего за своими пределами (не имеет побочных эффектов при рендеринге).
        
    - Если функция «грязная» (например, меняет глобальную переменную при рендере), двойной вызов сделает ошибку очевидной и визуально заметной.
        

### 2. Проверка эффектов (Effect Cleanup Check)

Особенно актуально для React 18+.

- **Что происходит:** При монтировании компонента React выполняет следующую цепочку:
    
    1. Монтирование (Mount) -> Запуск эффектов (`useEffect`).
        
    2. Размонтирование (Unmount) -> Запуск очистки эффектов (`cleanup function`).
        
    3. Повторное монтирование (Remount) -> Снова запуск эффектов.
        
- **Зачем:** Чтобы проверить, правильно ли вы написали функцию очистки (cleanup) в `useEffect`.
    
    - Если вы забыли удалить подписку, очистить таймер или закрыть соединение, этот цикл (Mount -> Unmount -> Mount) приведет к дублированию событий или утечке памяти, что сразу станет заметно.
        

### 3. Обнаружение устаревших API (Deprecation Checks)

React предупреждает о методах, которые считаются устаревшими и будут удалены в следующих версиях.

- **String Refs:** Использование строковых рефов (устаревший способ доступа к DOM).
    
- **findDOMNode:** Метод, который нарушает принципы абстракции компонентов.
    
- **Legacy Context:** Старый API контекста.
    

### 4. Предупреждение о неожиданных побочных эффектах

Strict Mode помогает найти побочные эффекты там, где их быть не должно:

- В теле функционального компонента.
    
- В функциях инициализации состояния (`useState(() => ...)`).
    
- В методах `constructor`, `render`, `shouldComponentUpdate` классовых компонентов.
___
### Важные нюансы

1. **Только Dev:** Все эти проверки (двойной рендер и т.д.) не работают в скомпилированном приложении (production). Пользователь не столкнется с замедлением.
    
2. **Можно использовать локально:** Не обязательно оборачивать все приложение. Можно обернуть только конкретную ветку компонентов, чтобы проверить только её.
